# Introduction

This document explains the engineering rationale behind the main architectural decisions adopted throughout the platform.

Rather than describing the implementation details of each component, it focuses on the design choices that support the functional and non-functional requirements of near real-time biomedical data processing. These decisions are derived exclusively from the project report and reflect the constraints, requirements, and trade-offs considered during the design process.

Where the report explicitly justifies a technology or architectural pattern, that rationale is documented here. Decisions that are not justified in the report are intentionally omitted.

---

# Decision 1: Serverless Architecture

## Context

The platform targets continuous processing of biomedical data generated by wearable devices in non-critical hospital environments. The system must process streaming data with low latency while remaining operationally simple and economically sustainable.

The report also identifies variability in workload as an important characteristic of the target workload. Different monitoring scenarios produce different event rates, requiring the platform to adapt without manual infrastructure management.

## Requirements

The selected architecture was required to satisfy several non-functional requirements defined during system design, including:

| Requirement            | Objective                                     |
| ---------------------- | --------------------------------------------- |
| Low latency            | Near real-time processing                     |
| Scalability            | Adapt to increasing event rates               |
| Operational simplicity | Minimize infrastructure management            |
| Cost efficiency        | Reduce operational costs                      |
| Fault tolerance        | Continue processing despite isolated failures |

## Selected Solution

The platform adopts a fully serverless architecture implemented using managed AWS services.

Processing, storage, monitoring, and messaging are delegated to managed cloud services, while infrastructure provisioning is performed through Infrastructure as Code.

## Rationale

According to the report, the serverless model aligns with the operational requirements of the platform by providing automatic scaling and reducing operational overhead.

Experimental evaluation demonstrates that the architecture absorbs a 26-fold increase in workload between the baseline and sustained scenarios without perceptible latency degradation or manual intervention. During these experiments, AWS Lambda automatically increased concurrent executions while maintaining stable processing performance.

The report also concludes that the operational cost remains consistent with the economic objectives of the platform, supporting the selection of the serverless paradigm for the evaluated workload.

## Advantages

The report explicitly identifies the following benefits:

* Automatic infrastructure scaling.
* Reduced operational complexity.
* Low operational cost.
* Managed cloud services.
* Near real-time processing under supported workloads.
* Infrastructure reproducibility through Infrastructure as Code.

## Trade-offs

The report discusses several trade-offs associated with the serverless model:

| Trade-off                 | Discussion                                                                    |
| ------------------------- | ----------------------------------------------------------------------------- |
| Reduced low-level control | The serverless model provides less control over low-level parallelism.        |
| Cloud provider dependency | The architecture depends on AWS managed services.                             |
| Cold starts               | Occasional cold starts are an inherent characteristic of the execution model. |

The report also notes that the single-region versus multi-region deployment strategy was not evaluated within the scope of the prototype.

## References

Related services:

* AWS Lambda
* Amazon Kinesis Data Streams
* Amazon DynamoDB
* Amazon S3
* Amazon SQS
* Amazon CloudWatch
* Terraform

# Decision 2: Event-Driven Architecture

## Context

Biomedical sensors continuously generate measurements at different sampling frequencies. The processing platform must ingest these events continuously while decoupling data producers from downstream processing components.

The architecture therefore requires a processing model capable of handling asynchronous event streams.

## Requirements

The report identifies the following design requirements for the ingestion layer:

| Requirement          | Objective                                  |
| -------------------- | ------------------------------------------ |
| Continuous ingestion | Accept uninterrupted event streams.        |
| Temporal ordering    | Preserve event ordering for each sensor.   |
| Scalability          | Support increasing event volumes.          |
| Decoupling           | Separate event generation from processing. |

## Selected Solution

The platform adopts an event-driven architecture in which biomedical measurements are processed as independent events transmitted through a streaming ingestion layer.

Each architectural component performs a single responsibility while communicating asynchronously through managed cloud services.

## Rationale

The report justifies this architecture as an appropriate model for continuous biomedical monitoring because it separates event generation, ingestion, processing, and persistence into independent stages.

This separation allows the ingestion layer to buffer incoming events while downstream processing scales independently.

Experimental results confirm that under baseline and sustained workloads the event-driven pipeline processes incoming events without accumulation, as indicated by an IteratorAge of zero throughout execution.

## Advantages

Benefits explicitly supported by the report include:

* Separation between producers and consumers.
* Independent scaling of processing.
* Continuous stream processing.
* Preservation of event ordering through partition keys.
* Buffering of temporary workload variations.

## Trade-offs

The report identifies one important architectural trade-off related to event partitioning.

The selected partition key preserves temporal ordering for each sensor but produces an uneven distribution of events across Kinesis shards when sensors generate data at significantly different frequencies.

This imbalance limits the effective parallelism of downstream processing under extreme workloads.

## References

Related services:

* Amazon Kinesis Data Streams
* AWS Lambda

# Decision 3: Amazon Kinesis Data Streams

## Context

The platform requires an ingestion service capable of receiving continuous biomedical event streams before processing them asynchronously.

The selected technology also needed to support buffering between event producers and consumers.

## Requirements

| Requirement         | Objective                           |
| ------------------- | ----------------------------------- |
| Streaming ingestion | Receive continuous event streams.   |
| Event buffering     | Decouple ingestion from processing. |
| Ordering            | Preserve temporal ordering.         |
| Scalability         | Support increasing workloads.       |

## Selected Solution

Amazon Kinesis Data Streams is used as the streaming ingestion layer between the biomedical simulator and the processing pipeline.

## Rationale

The report identifies Kinesis as the component responsible for decoupling event production from event processing.

Experimental analysis also reveals that Kinesis is the primary determinant of end-to-end latency. Under baseline and sustained workloads, IteratorAge remains at zero, indicating that events are consumed without backlog.

During the burst scenario, however, Kinesis becomes the limiting component of the architecture. The report attributes the observed degradation to throttling at the ingestion layer caused by the configured shard capacity and the non-uniform distribution of partition keys.

Importantly, the report concludes that the processing logic executed by Lambda is not the bottleneck. Instead, latency accumulates while events remain buffered inside Kinesis waiting to be consumed.

This experimental diagnosis constitutes one of the principal engineering findings of the project.

## Advantages

The report explicitly attributes the following benefits to the selected solution:

* Asynchronous buffering of incoming events.
* Preservation of temporal ordering.
* Decoupling between producers and consumers.
* Stable operation under baseline and sustained workloads.

## Trade-offs

The report identifies several limitations associated with the evaluated configuration.

| Trade-off                  | Discussion                                                                                                                |
| -------------------------- | ------------------------------------------------------------------------------------------------------------------------- |
| Shard capacity             | Five shards were insufficient under burst workloads.                                                                      |
| Partition key distribution | Uneven event distribution reduced effective throughput.                                                                   |
| Backlog recovery           | The evaluated configuration required approximately 3.4 minutes to drain the accumulated backlog after the burst scenario. |

These limitations are presented as characteristics of the evaluated infrastructure configuration rather than limitations of the architectural pattern itself.

# Decision 4: AWS Lambda

## Context

The platform requires a compute layer capable of processing continuous streams of biomedical events without provisioning or managing servers. This processing stage is responsible for transforming incoming events before persisting them in the storage layer while maintaining near real-time performance.

The compute layer also needed to scale automatically as ingestion rates increased.

## Requirements

| Requirement              | Objective                                                      |
| ------------------------ | -------------------------------------------------------------- |
| Event processing         | Process every accepted event automatically.                    |
| Automatic scaling        | Adapt execution capacity to workload.                          |
| Low operational overhead | Eliminate server administration.                               |
| Data persistence         | Store processed and raw data in their respective repositories. |
| Observability            | Publish operational metrics for monitoring.                    |

## Selected Solution

The platform uses AWS Lambda as the processing engine of the event pipeline.

Lambda is invoked directly from Amazon Kinesis Data Streams through an Event Source Mapping and executes all processing logic before persisting the resulting data.

## Rationale

The report identifies Lambda as the processing component responsible for implementing the business logic while benefiting from automatic scaling.

Experimental results demonstrate that Lambda scales from a maximum of three concurrent executions in the baseline scenario to eleven concurrent executions during sustained and burst workloads without operational intervention.

An important outcome of the experimental evaluation is that Lambda is **not** identified as the system bottleneck. Throughout all scenarios, processing latency remains negligible compared with the time events spend waiting inside the Kinesis buffer. Even when storage operations are considered, Lambda execution time remains substantially lower than the observed pipeline latency.

The report therefore concludes that increasing Lambda memory or internal processing parallelism alone would not resolve the latency degradation observed during burst workloads.

## Advantages

The report explicitly identifies the following benefits:

* Automatic scaling according to workload.
* Zero processing errors during all experimental scenarios.
* Successful execution under baseline, sustained, and burst workloads.
* Integration with the monitoring infrastructure through CloudWatch metrics.
* Simultaneous persistence of raw and processed data.

## Trade-offs

The report discusses one relevant design consideration.

Although Lambda was configured with a maximum theoretical concurrency derived from the number of Kinesis shards and the configured `parallelization_factor`, the observed concurrency remained well below this limit because event distribution across shards was uneven.

Consequently, the effective processing parallelism was constrained by the ingestion layer rather than by Lambda itself.

* Amazon S3

# Decision 5: Amazon DynamoDB

## Context

The processed biomedical events require persistent storage capable of supporting continuous writes generated by the streaming pipeline.

The storage layer must integrate naturally with the serverless architecture while maintaining consistent performance under the evaluated workloads.

## Requirements

| Requirement              | Objective                                        |
| ------------------------ | ------------------------------------------------ |
| Persistent storage       | Store processed biomedical records.              |
| Low operational overhead | Use a managed storage service.                   |
| Scalability              | Support continuously increasing write workloads. |
| Integration              | Operate directly from the processing pipeline.   |

## Selected Solution

The platform stores processed biomedical events in Amazon DynamoDB.

## Rationale

The report selects DynamoDB as the operational datastore for processed events within the serverless architecture.

Experimental evaluation confirms that DynamoDB operates without observable performance degradation in every evaluated scenario. The report also highlights that data integrity remained complete in both the baseline and sustained workloads, where every generated event was successfully stored.

The storage layer therefore did not contribute to the latency or throughput limitations identified during the experiments.

## Advantages

The report explicitly attributes the following benefits to the selected solution:

* Stable behaviour throughout all evaluated scenarios.
* Successful integration with the serverless processing pipeline.
* Support for the platform's data integrity objectives.
* Managed operation without infrastructure administration.

## Trade-offs

The report does not explicitly discuss architectural trade-offs associated with selecting Amazon DynamoDB.

The experimental analysis instead concludes that the limitations observed during burst workloads originated in the ingestion layer rather than in the persistence layer.

# Decision 6: Amazon S3

## Context

In addition to processed biomedical records, the platform requires preservation of the original event stream.

Maintaining an immutable copy of the incoming data enables future inspection and potential reprocessing without depending exclusively on the processed representation.

## Requirements

| Requirement             | Objective                            |
| ----------------------- | ------------------------------------ |
| Raw data preservation   | Store the original incoming events.  |
| Independent persistence | Separate raw and processed datasets. |
| Managed storage         | Avoid infrastructure management.     |

## Selected Solution

Amazon S3 is used to store the raw biomedical events produced by the processing pipeline.

## Rationale

The report adopts dual persistence, separating processed records from their original representations.

Within this design, Amazon S3 provides long-term storage for the raw event stream while Amazon DynamoDB stores the processed records used by the platform.

The report presents this separation as an architectural decision that preserves the original dataset independently of subsequent processing.

## Advantages

Benefits explicitly supported by the report include:

* Independent storage of raw data.
* Preservation of original event records.
* Integration with the serverless architecture.

## Trade-offs

The report does not explicitly discuss trade-offs associated with selecting Amazon S3.

# Decision 7: Amazon SQS Dead-Letter Queue

## Context

The platform requires a mechanism for isolating events that cannot be processed successfully without interrupting the remaining data pipeline.

## Requirements

| Requirement            | Objective                                                           |
| ---------------------- | ------------------------------------------------------------------- |
| Failure isolation      | Prevent individual processing failures from affecting the pipeline. |
| Error traceability     | Preserve failed events for later analysis.                          |
| Operational robustness | Improve fault tolerance of the processing workflow.                 |

## Selected Solution

The architecture incorporates an Amazon SQS Dead-Letter Queue (DLQ) connected to the processing pipeline.

## Rationale

The report includes a Dead-Letter Queue as part of the platform's fault-tolerance strategy.

During the experimental evaluation, the DLQ remained empty across all scenarios because Lambda completed processing without execution errors.

The report further explains that the event loss observed during the burst scenario occurred before Lambda processing, as a consequence of throttling in the Kinesis ingestion layer. Since those rejected events never reached Lambda, they were not routed to the DLQ.

This distinction allowed the experimental analysis to identify the ingestion layer, rather than the processing layer, as the source of data loss.

## Advantages

The report explicitly supports the following benefits:

* Isolation of processing failures.
* Preservation of failed events for inspection.
* Contribution to the platform's fault-tolerance strategy.
* Confirmation that Lambda processing completed without execution errors during the experiments.

## Trade-offs

The report does not identify any trade-offs associated with the adoption of the Dead-Letter Queue.

Although future work proposes automatic DLQ reprocessing, this functionality is intentionally outside the scope of the current platform.

Related services:

* AWS Lambda
* Amazon Kinesis Data Streams
* Amazon CloudWatch

# Decision 8: Amazon CloudWatch

## Context

The platform requires operational visibility throughout the experimental evaluation to verify that the processing pipeline satisfies its latency, throughput, and reliability objectives.

Monitoring was also necessary to identify bottlenecks and validate the behavior of the managed services under different workloads.

## Requirements

| Requirement            | Objective                                               |
| ---------------------- | ------------------------------------------------------- |
| Operational monitoring | Observe the behavior of the processing pipeline.        |
| Performance evaluation | Collect quantitative metrics for experimental analysis. |
| Error detection        | Monitor failures during execution.                      |
| Scalability analysis   | Evaluate service behavior under increasing workloads.   |

## Selected Solution

Amazon CloudWatch is used as the primary monitoring service for the platform.

## Rationale

The report relies extensively on CloudWatch metrics during the experimental evaluation.

Metrics such as **IteratorAge**, Lambda concurrency, invocation duration, and error rate provide the evidence used to characterize the behavior of the architecture under baseline, sustained, and burst workloads.

CloudWatch measurements were fundamental to identifying that latency accumulated within the Kinesis buffer rather than during Lambda execution. The collected metrics also confirmed that Lambda maintained a zero error rate throughout all experiments while scaling automatically according to workload.

## Advantages

The report explicitly supports the following benefits:

* Continuous operational monitoring.
* Quantitative performance measurements.
* Identification of system bottlenecks.
* Verification of latency and scalability objectives.
* Support for the experimental evaluation methodology.

## Trade-offs

The report does not explicitly discuss architectural trade-offs associated with Amazon CloudWatch.

# Decision 10: Terraform (Infrastructure as Code)

## Context

The platform requires a deployment approach capable of provisioning the complete cloud infrastructure consistently and reproducibly.

The infrastructure also needed to be versioned together with the application source code.

## Requirements

| Requirement     | Objective                                                  |
| --------------- | ---------------------------------------------------------- |
| Reproducibility | Deploy identical infrastructure across environments.       |
| Automation      | Eliminate manual infrastructure provisioning.              |
| Version control | Maintain infrastructure definitions alongside source code. |

## Selected Solution

The platform provisions its cloud infrastructure using Terraform.

## Rationale

The report adopts Infrastructure as Code to automate deployment of all cloud resources.

Terraform enables the complete infrastructure to be recreated from declarative configuration files, improving deployment consistency and simplifying future modifications.

The report also notes that Infrastructure as Code would facilitate future deployment extensions, including multi-region architectures.

## Advantages

Benefits explicitly identified in the report include:

* Reproducible infrastructure deployment.
* Automated provisioning.
* Infrastructure versioning.
* Simplified future infrastructure evolution.

## Trade-offs

The report does not explicitly discuss trade-offs associated with selecting Terraform.

# Architectural Principles

The engineering decisions collectively reflect a consistent architectural approach focused on the non-functional requirements defined for the platform.

| Principle                  | Evidence from the report                                                                                                                                                                                               |
| -------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Scalability**            | Managed services automatically adapt to increasing workloads, with Lambda scaling automatically and the platform sustaining a 26× increase in event rate without latency degradation in the supported operating range. |
| **Decoupling**             | Streaming ingestion separates event generation from downstream processing through an event-driven pipeline.                                                                                                            |
| **Fault Tolerance**        | Failed processing can be isolated through a Dead-Letter Queue, while managed services continue operating independently.                                                                                                |
| **Operational Simplicity** | The architecture relies on managed AWS services and Infrastructure as Code to reduce operational effort.                                                                                                               |
| **Cost Efficiency**        | The report concludes that the operational costs observed during experimentation are consistent with the adopted serverless model.                                                                                      |
| **Observability**          | CloudWatch metrics provide the operational visibility required to evaluate latency, throughput, scalability, and system behaviour.                                                                                     |

# Key Takeaways

The engineering decisions consistently prioritize managed services, event-driven processing, and Infrastructure as Code to satisfy the operational objectives of the platform while minimizing infrastructure management.

The experimental evaluation demonstrates that the architecture performs reliably within its intended operating range and provides sufficient observability to diagnose performance limitations. Rather than identifying shortcomings in the processing logic, the results show the importance of infrastructure configuration, particularly within the ingestion layer, for achieving scalability objectives.

Together, these decisions establish a modular, reproducible, and operationally simple platform for near real-time biomedical data processing.
